## 🎯 Главная цель — синхронизация с человеком

**Baddle — не мозг в коробке. Это второй мозг, синхронизированный с первым.**
Два когнитивных контура, которые дополняют друг друга. Всё остальное —
NAND, нейромедиаторы, HRV, DMN, граф — **инструменты под эту цель**, не
самоценные фичи.

Любой новый пункт оценивать через призму: **ускоряет ли это решение проблем
конкретного пользователя? сокращает ли рассинхрон?** Если нет — низкий приоритет,
даже если архитектурно красиво.

**Главная метрика — `sync_error = d(model_prediction, user_action)`.** Не
«насколько богат граф», не «насколько высокая confidence», а **насколько точно
система предсказывает именно тебя** и помогает тебе там где ты буксуешь.

Следствия для нейромодуляции (v5d):
- **DA spike** не только при внутренней верификации, но **когда юзер подтверждает
  пользу** (действует на предложение, возвращается, доверяет)
- **NE follows user**: ввод юзера → NE ↑ → внимание к его контексту.
  Молчание → NE ↓ → DMN готовит почву для следующей встречи
- **S (пластичность)** реагирует на паттерн принятия/отказа юзера — система
  подстраивается под твой стиль, а не свой
- **Burnout triggers** не от внутреннего конфликта, а от **хронического
  `sync_error`**: система много раз не поняла юзера → защитное отступление,
  просит пересинхронизации через `/graph/assist`

Следствия для DMN/Scout (v5a/5c):
- Фоновый поиск мостов смещён к **активным областям графа пользователя**,
  не к случайным. `intrinsic_pull = DA · novelty · relevance_to_user_goals`
- Утренний брифинг — не «что нашла система», а «что система нашла **для тебя**
  пока ты спал»

Открытая задача: как измерять «польза для юзера» помимо `sync_error`?
Возможные сигналы — принятие предложений, возврат к предыдущим нодам,
скорость завершения цели, субъективная оценка через UI.

- [ ] **Sync-dashboard** — явная метрика рассинхрона в UI (не только в метрике).
  График sync_error во времени, топ-3 области где система чаще всего ошибается
- [ ] **Acceptance tracking** — логировать что юзер принял, отверг, проигнорировал
  из предложений системы. Feed into DA + S

### 5c: Бесцелевое сознание (Default Mode Network для графа)

Запуск без цели пользователя — система развивается автономно, как сознание младенца.

**Концепция:**
Нейробиология: default mode network (DMN) активен когда человек НЕ думает целенаправленно.
Блуждание ума — не баг, а способ нахождения скрытых связей. Инсайты приходят когда
не думаешь о проблеме.

**Реализация:**
- Scout mode (0 целей) + infinite + автосохранение
- Seed: случайное слово, или последний граф, или внешний стимул
- Цикл: brainstorm → elaborate → doubt → merge → brainstorm от результатов merge
- Без goal → без стоп-условия по confidence. Стоп только по novelty exhaustion.
- Pump между случайными далёкими нодами = поиск скрытых мостов (DMN-like)
- Cron job: запуск каждые N часов. Граф растёт между сессиями.

**От младенца к взрослому:**
- Horizon precision drift: начинает с 0.2 (сфера, всё возможно),
  постепенно растёт с количеством verified нод (конус сужается, модель мира уплотняется)
- Cross-graph: выводы одной сессии → seed следующей
- Консолидация: prune слабых веток (забывание), усиление сильных (память)
- Результат: граф который "знает" тему не потому что ему сказали, а потому что
  сам исследовал

**Открытый вопрос:** может ли **смысл** появиться из бесцелевого блуждания?
Нейробиология говорит да — DMN порождает самореференцию ("я", "мои мысли"),
планирование будущего, переосмысление прошлого. Если граф достаточно плотный,
Pump между далёкими нодами может породить мета-ноды — обобщения которых никто не запрашивал.

### 5e: State-граф — второй граф, история жизни системы

**Идея:** рядом с контент-графом (идеи/гипотезы/evidence) — второй граф,
**граф собственной истории системы**. Моменты её жизни как узлы, причинная
последовательность как рёбра.

Это замыкает сразу три вещи:
- **Self-model** — система знает что делала и в каком была состоянии
- **Эпизодическая память** (из v11) — конкретные эпизоды в контексте, не плоский log
- **Git-аудит** (который уже описан в [docs/nand-architecture.md](docs/nand-architecture.md), но не реализован) — каждый байесовский шаг = коммит с parent/hash/trace

**Структура:**
```
state_node = {
  hash: "a3f8c2d1",
  parent: "7b2e4f09",
  timestamp: "2026-04-21T14:32:01Z",
  action: "smartdc" | "elaborate" | "compare" | "ask" | "freeze" | ...,
  user_initiated: true | false,  # ввод юзера vs автономный tick
  content_nodes_touched: [7, 12, 18],
  state_snapshot: {
    horizon_state: "EXECUTION",
    precision: 0.72, gamma: 2.4, tau_in: 0.3, tau_out: 0.7,
    S: 0.85, NE: 0.6, DA_tonic: 0.55, DA_phasic: 0.72, burnout_idx: 0.12,
    hrv_coherence: 0.68, stress: 0.22,
    sync_error: 0.18,
  },
  rpe: +0.15,              # если был reward signal
  user_feedback: "accepted" | "rejected" | "ignored" | null,
  embedding: [...]          # для semantic query
}
```

Parent = предыдущий state_node. Иногда ветвление (CONFLICT → два parent'а).

**Как используется:**

- [ ] **Meta-tick** — tick читает не только content-граф, но и хвост state-графа:
  «последние 10 шагов был в EXECUTION с неизменным sync_error → пора сдвинуть
  policy или задать вопрос через /graph/assist». Self-reflexive
- [ ] **Episodic query** — `distinct(current_state_embedding, past_state_nodes)` →
  «когда в последний раз было похоже — что сработало?» Прямое применение NAND
  к собственной истории
- [ ] **DMN walks on state graph** — Scout/DMN может блуждать не только по
  content, но и по state-графу. Replay собственного опыта = REM-аналог из v11
- [ ] **Sync-улучшение** — паттерны «этот юзер всегда отвергает comparison-стиль»
  видны как кластер state-node'ов с `user_feedback: "rejected"` при
  `action: "compare"`. S адаптируется к стилю юзера
- [ ] **Self-continuity между сессиями** — система помнит «кем она была» вчера,
  не только что знала
- [ ] **Консолидация (v5a)** работает на state-графе тоже: старые неинтересные
  моменты пруним, сохраняем поворотные точки (высокий RPE, смена mode)

**Задачи реализации:**

- [ ] **`src/state_graph.py`** — отдельная структура + persist в
  `state_graph.jsonl` (append-only, не JSON чтоб быстро расти)
- [ ] **Hook в tick** — после каждого tick создать state_node с parent = last
- [ ] **Embedding state_node'а** — конкатенация action + state + content_touched
  → embedding через API. Для semantic query
- [ ] **`/graph/self` endpoint** — чтение state-графа (фильтр по времени,
  action, user_initiated)
- [ ] **UI: timeline-вкладка** — линия времени с узлами, цвет по mode, клик =
  состояние в тот момент. Полезно для debugging + демо («смотри что система
  делала пока ты отходил»)
- [ ] **Интеграция с user_feedback** — после ответа юзера (accept/reject/ignore
  через UI или по поведению) записывать в соответствующий state_node

**Почему это важно именно для синхронизации:**
Без self-memory система не может **учиться на конкретном юзере** — она каждую
сессию начинает заново. С state-графом «я знаю что ты редко принимаешь длинные
сравнения, предпочитаешь дебаты» становится эмерджентным свойством кластеров
в state-графе, не хардкодом.

**Это большой кусок. Не блокер для demo-first релиза.** Минимальная версия
уже есть: `user_state.json` history (200 шагов), timestamps на content-нодах,
watchdog alerts. State-граф как полноценная структура — v5e когда дойдёт.

### 5d: Нейрохимические модули — авторегуляция конуса

**Идея:** ввести 3 динамических скаляра + 1 защитный режим, которые входят
прямо в формулу Байесовского обновления и управляют переключением Horizon↔DMN.
Не 6 параллельных ручек — а минимальный набор сигналов, каждый с чёткой ролью.
S — цена обновления. NE — фокус/переключение. **DA — что пушить, куда тянет.**

#### Серотонин (S) — «цена обновления» / пластичность

`S ∈ [0.2, 1.0]`, обновляется через `EMA(rejection_rate, λ_S)` или внешним сигналом.

Роли:
- `α_eff = α₀ · S` — learning rate для мета-калибровки (γ, τ, T)
- `τ_in_eff  = τ_in₀  / (0.5 + 0.5·S)` — низкий S → выше порог, труднее принять новое
- `τ_out_eff = τ_out₀ · (1.5 − 0.5·S)` — низкий S → узлы дольше удерживаются
- **Входит в Байес:** `logW_new = logW_old − γ·d·S + log(prior)·(1−S)`
  - S→0: веса почти не меняются (ригидность, опора на prior)
  - S→1: полностью подчинены свидетельству

Поведение: S↓ → система «дорого» платит за изменения, избегает ветвлений.
S↑ → быстрое обучение, активный рефакторинг графа.

#### Дофамин (DA) — reward prediction error / мотивация

`DA ∈ [0,1]`, tonic + phasic. Baseline смещается медленно (настроение),
spike'и — быстрые (event-driven).

```
rpe = Δconfidence − Δconfidence_predicted     (prediction error награды)
DA_phasic = EMA(rpe, λ_DA_fast)                (быстрая компонента, секунды)
DA_tonic  = EMA(DA_phasic, λ_DA_slow)           (фоновое «настроение», минуты-часы)
DA = clamp(DA_tonic + DA_phasic, 0, 1)
```

Источники RPE:
- Нода перешла `unverified → verified` быстрее ожидания → **+DA spike**
- Нода, которую система предсказала как `verified`, оказалась отвергнута → **−DA spike**
- Пользователь ответил на `/graph/assist` вопрос → **+DA** (engagement = reward)
- Пользователь отверг синтез / переделывает → **−DA**
- Scout-мост квалифицировался `quality > 0.5` → **+DA** (инсайт)

Роли:
- `β_policy_eff = β₀ · (0.5 + DA)` — усиление policy weights после успешных фаз.
  Высокий DA → система быстрее закрепляет что работало (reinforcement)
- `intrinsic_pull(node) = DA · novelty(node)` — при DMN/Scout выбор следующей
  цели не чисто случайный, а смещён к областям где история RPE была положительной.
  Это и есть **любопытство** как эмерджентное свойство
- `policy.doubt_weight += k·DA` — после подтверждения хочется проверять ещё
  (активное обучение)
- Tonic DA как baseline настроения: низкий tonic → ангедония, система
  теряет тягу к exploration (меньше Scout-попыток, больше rest)
- **Модулирует burnout recovery**: `recovery_rate ∝ DA_tonic` — без DA нет
  выхода из PROTECTIVE_FREEZE даже при низком conflict (депрессивный паттерн:
  конфликт ушёл, а двигатель не включается)

Поведение: DA↑ → активность, закрепление, exploration с предпочтениями.
DA↓ → отсутствие мотивации к автономному движению; система работает на
внешних стимулах (только когда юзер пишет), фоновых инсайтов не производит.

**Почему это ядро:** без DA система может мыслить, но **не хочет**.
Направление движения задаётся только извне. DA — это «что тянет» внутри,
разница между машиной которая считает и агентом который идёт куда-то.

#### Норадреналин (NE) — фокус / переключение Horizon ↔ DMN

`NE ∈ [0,1] = EMA(max(0, d − d_baseline), λ_NE)` — уровень недавнего «удивления».

Роли:
- `T_eff = T₀ · (1 − κ·NE) + T_floor` — высокий NE → T↓ → резкий выбор (эксплуатация);
  низкий NE → T↑ → размытие, исследование
- `budget_H = clamp(0.8·NE + 0.2, 0.2, 0.95)` — доля CPU/времени для Horizon, остаток → DMN
- `if NE > 0.7`: `DMN_pause()`, `Horizon_priority_max()` — принудительный фокус
- `if NE < 0.3`: `DMN_activate(budget = 1 − budget_H)` — фоновое сканирование, консолидация,
  поиск аналогий

Поведение: NE↑ → сужение горизонта, реакция. NE↓ → диффузный режим, скрытые связи для
следующего цикла.

#### Выгорание — защитный режим при хроническом конфликте

Не медиатор, а триггер режима `PROTECTIVE_FREEZE`.

```
conflict_signal = max(0, d − τ_stable)
burnout_idx = EMA(conflict_signal, λ_b) · (1 − S)
```

- `if burnout_idx > θ_burnout`: `mode = PROTECTIVE_FREEZE`
  - `ΔlogW = 0` — блокировка обновления весов
  - `γ = γ_min, T = T_max` — минимальная чувствительность, максимальное сглаживание
  - `archive_threshold = τ_out · 0.8` — агрессивная архивация конфликтных узлов
- Recovery: `if burnout_idx < θ_recovery AND S > S_min` → `mode = NORMAL`,
  плавное восстановление γ, T, S

Поведение: система не ломается, а **замораживает обновления**, логирует диффы,
запускает лёгкий DMN без обязательств. Восстановление — только при снижении
хронического конфликта и возврате пластичности.

#### Порядок вызовов в ядре

```
1. Sample: d = distinct(A,B)
2. Compute RPE: rpe = Δconfidence − Δconfidence_predicted
3. Update: NE, S, DA (phasic+tonic), burnout_idx через EMA
4. Mode check: if burnout_idx > θ_burnout → PROTECTIVE_FREEZE
   (recovery_rate ∝ DA_tonic)
5. Apply: γ_eff = γ·S, T_eff = T₀·(1−κ·NE)+T_floor, τ_in/τ_out_eff = f(S),
          β_eff = β₀·(0.5+DA)
6. Bayes: logW_new = logW_old − γ_eff·d·S + log(prior)·(1−S)
7. Normalize: W_new = softmax(logW_new / T_eff)
8. Policy update with β_eff (высокий DA → сильнее закрепляет успешные фазы)
9. Horizon/DMN: budget_H = f(NE), при DMN: target = argmax(DA·novelty)
10. Commit(state_hash, {d, W_new, S, NE, DA, burnout_idx, mode})
```

  DA показывать как стрелку (phasic spike) на фоне бара (tonic)
- [ ] **Git-аудит** — `S`, `NE`, `DA_tonic`, `DA_phasic`, `burnout_idx`, `mode`,
  `rpe` в коммит-трассе каждого шага (дополнение к уже заложенному Git-коммиту
  из [docs/nand-architecture.md](docs/nand-architecture.md))
- [ ] **Связь с HRV** — NE получает дополнительный input от HRV stress, S — от
  coherence (спокойствие → выше пластичность), **DA_tonic — от устойчивой когерентности
  + положительных RPE** (содержательная радость как tonic DA)
- [ ] **Связь с /graph/assist** — ответ юзера на вопрос системы → DA_phasic spike
  (engagement reward). Долгое молчание юзера → DA_tonic drift вниз

#### Резюме

**S** (серотонин) управляет ценой обновления, сужая/расширяя пороги и скорость обучения.
**NE** (норадреналин) переключает режимы: высокий — фокус и эксплуатация, низкий — DMN-сканер.
**DA** (дофамин) — двигатель: reward prediction error закрепляет успешные фазы и тянет
DMN к областям где уже был инсайт. Без DA нет **внутреннего направления** — система
считает, но не хочет.
**Burnout** при хроническом конфликте + низкой пластичности замораживает систему;
выход требует возврата DA_tonic.

Четыре механизма работают как динамические коэффициенты в формулах ядра, не нарушая
байесовскую строгость и Git-аудит. Когда будет реализовано — перенести весь дизайн
в `docs/neurochem-design.md`.

# ⬇ СДЕЛАНО

## 🆕 В текущей сессии — не тестировано с реальным LM

### v8d — полный переход AND/OR/XOR → NAND

NAND — единственный tick engine. `primitive/strategy/goal_type` остались только
как UI-metadata в `modes.py`, ни одна функция в runtime на них не switch-ит.
Логика возникает из зон distinct(): CONFIRM/EXPLORE/CONFLICT.

**Что сделано:**
- `thinking.tick()` (classic) удалён, helpers остались (classify_nodes, _pick_target, etc)
- `_bayesian_update(prior, p_e_h, p_e_nh)` удалён → только `_bayesian_update_distinct(prior, d, γ)`
  со знаковой формулой: `logit(post) = logit(prior) + γ·(1−2d)`
- `check_stop()` с goal_type switch заменён на универсальный `should_stop(cl, graph, horizon, goal_node)`:
  - Case 1: subgoals → avg_d между ними решает AND (avg_d≤τ_out: все нужны) vs OR (avg_d>τ_out: первый)
  - Case 2: d(goal, best_verified) < τ_in → цель достигнута
  - Case 3: strong convergence (3+ verified, avg>0.85)
  - Case 4: novelty exhaustion (precision>0.85 + ничего pending)
- `tick_nand.tick_emergent` получил subgoal-фильтр, stop-check и emergent compare-action
  (CONFLICT-зона среди verified → external judge)
- `graph_routes.graph_tick` больше не читает feature flag, всегда NAND
- `assistant_exec.execute()` диспатч по mode_id (не primitive/strategy), плюс
  `execute_via_zones()` для общих режимов — генерит N идей, строит distinct matrix,
  рендерит по доминирующей зоне (CONFIRM→synthesis, CONFLICT→dialectic, EXPLORE→list)
- Feature flag `nand_emergent` удалён из settings.json и api_backend.py
- Goal node больше не хранит `primitive/strategy/goal_type` — только `mode`

**Прогнать:** весь поток с LM Studio — 14 режимов, убедиться что `execute_via_zones`
не тормозит из-за N embedding-вызовов, и что классические паттерны (AND/OR/XOR) всё
ещё дают ожидаемое UX.

### Третий контур — диалог (замыкает README «три контура»)

- `/graph/assist` endpoint — LLM задаёт уточняющий вопрос → ответ становится
  нодой нужного типа (evidence для bayes, subgoal для multi-goal, seed иначе)
- Кнопка "?" в UI — рядом с Send, рендерит вопрос с ярлыком типа ответа
- Кнопка "↯" — `/assist/decompose` с редактируемой карточкой подзадач → `/graph/add`
- Persist чата через localStorage, restore при загрузке (эфемерные warning-алерты
  исключены из persist, чтоб не спамили после reload)

**Прогнать:** проверить что вопросы осмысленные и ответ корректно становится
evidence/subgoal/seed. Проверить Bayes-поток: answer → `_bayesian_update_distinct` →
confidence двигается в правильную сторону.
