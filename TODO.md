## v1.5: CognitiveHorizon — адаптивный движок мышления

Динамический контроллер, который регулирует параметры генерации на основе обратной
связи. Не генерирует контент — управляет **как** генерировать. Сидит между tick() и
LLM-вызовами, адаптируя temperature/top_p/prompts по ходу цикла.

### Проблема сейчас

tick() использует **фиксированные** параметры: temp=0.9, top_k=40 на все фазы.
Generate, Doubt, Collapse — всё с одинаковой температурой. Это как ехать на одной
передаче: можно, но неэффективно.

### Решение: три параметра Horizon

```
Π (precision)       : 0.0–1.0. Уверенность в текущей модели.
                      Выше = уже фильтр гипотез = ниже температура.
Λ (policy_weights)  : {generate: 0.3, merge: 0.2, elaborate: 0.2, doubt: 0.3}
                      Веса фаз. Определяют что делать на текущем шаге.
Γ (context_frame)   : Активная рамка — промпт, правила, ограничения.
                      Переключается при высокой novelty.
```

### Правила обновления

```
После каждого шага tick:

1. Precision:
   P_new = clamp(P_old + α * (surprise - target_surprise), 0.0, 1.0)

   surprise = 1 - confidence_после_SmartDC   (или 1 - cosine(expected, actual))
   target_surprise = гиперпараметр по задаче (0.15–0.6)
   α = скорость адаптации (0.05–0.2)

   Если surprise > target → precision падает → температура растёт → шире ищем.
   Если surprise < target → precision растёт → температура падает → фокусируемся.

2. Policy Weights:
   W_i += β * gradient_i
   W = normalize(W)

   gradient = +1 если фаза дала хороший результат (confidence вырос),
              -1 если провал (confidence упал или novelty reject)
   β = шаг обновления (0.1–0.3)

   Фазы которые работают хорошо → их вес растёт → tick выбирает их чаще.

3. Context Frame:
   if novelty > threshold: switch_frame()

   novelty = 1 - max_similarity(new_output, last_N_outputs)
   Высокая novelty = модель выдала что-то непохожее на предыдущее → сменить рамку.
```

### Маппинг Precision → LLM параметры

```python
temperature    = clamp(1.0 - precision, 0.1, 1.5)
top_p          = clamp(0.5 + 0.5 * precision, 0.7, 0.95)
presence_pen   = 0.0 if precision > 0.6 else 0.5
```

| Precision | Temperature | Режим мышления |
|-----------|-------------|---------------|
| 0.3 | 0.7 | Широкий поиск, креативность (GENERATE, META) |
| 0.5 | 0.5 | Сбалансированный (ELABORATE, COLLAPSE) |
| 0.8 | 0.2 | Узкий фокус, точность (DOUBT/SmartDC) |
| 0.9 | 0.1 | Детерминированный (финальный синтез) |

### Четыре состояния Horizon

| Состояние | Precision | Когда в нашем цикле | Триггер |
|-----------|-----------|--------------------|---------| 
| **EXPLORATION** | 0.3–0.5 | GENERATE, META | Мало гипотез, высокая entropy |
| **EXECUTION** | 0.7–0.9 | DOUBT, финальный COLLAPSE | Есть фокус, проверяем |
| **RECOVERY** | 0.4→0.6 | После провала SmartDC, collapse rollback | Surprise spike |
| **INTEGRATION** | 0.5–0.6 | MERGE, ELABORATE, промежуточный COLLAPSE | Данные собраны, синтезируем |

Переходы автоматические: surprise spike → RECOVERY, всё verified → EXECUTION,
META round → EXPLORATION, merge → INTEGRATION.

## v1.6: Накачка (Pump) — поиск третьего элемента между мыслями

Новый примитив работы с узлами, параллельный Smart DC. Две мысли A и B не имеют прямой
связи. Накачка ищет **третий элемент C** — промежуточное понятие, которое связано с обеими.
C не обязательно похоже на A или B по смыслу: оно дополняет их до общего контекста.

### Суть

**Smart DC** принимает одну ноду и проверяет её через диалектику (тезис/антитезис/синтез → центроид → confidence).

**Накачка** принимает две ноды и ищет **третий элемент** (мост) через двустороннюю
экспансию: каждая нода генерирует вокруг себя облако ассоциаций. В какой-то момент
в одном из облаков (или в обоих) появляется концепт C, который связан и с A, и с B.
Это не "A встречается с B", это "найдено C, которое делает A и B связанными".

### Физическая аналогия

Частицы соединяются если сумма спинов = 360. A = 180, B = 90, вместе = 270. Не хватает.
Добавляем 90 (третий элемент C) → 180 + 90 + 90 = 360 → связь замкнулась.

Важно: **C не похож на A или на B**. 90 не похож на 180, и 90 не обязательно похож
на другие 90. Важна не похожесть, а то что C **дополняет недостающее** — связывает
концы, которые сами по себе разомкнуты.

Другой пример: A = "иммунная система", B = "корпоративные банкротства". Они не
похожи. Но третий элемент C = "адаптация к внешним угрозам" связан и с иммунитетом,
и с бизнесом. C — мост, потому что он относится к обоим, хотя сам по себе не похож
ни на "иммунитет", ни на "банкротство".

### Отличие от поиска пути (BFS / shortest path)

BFS находит *существующие* связи в текущем графе. Если пути нет — результат "нет
связи". Накачка наоборот: если пути нет — **генерирует** промежуточные узлы пока
не появится третий элемент C. Это не поиск, это **создание моста**.

### Алгоритм

```
Вход: нода A, нода B

Инициализация:
    cloud_A = {A} + (уже существующие соседи A в графе)
    cloud_B = {B} + (уже существующие соседи B в графе)

Цикл (max_iterations = 5):
    1. Расширить cloud_A:
       - elaborate: что следует из A? (механизмы, детали, причины)
       - expand: какие есть аспекты A? (разные углы рассмотрения)
       - свободные ассоциации с контекстом A
       → новые узлы добавляются в cloud_A
    2. Расширить cloud_B аналогично

    3. Поиск скрытого параметра (LLM-as-abstractor):
       Вызов LLM с промптом:
           "У меня две группы понятий, связанные с двумя разными идеями.

            Идея A: [text(A)]
            Облако контекста A:
              - [text(a_1)]
              - [text(a_2)]
              - ...

            Идея B: [text(B)]
            Облако контекста B:
              - [text(b_1)]
              - [text(b_2)]
              - ...

            Вопрос: какой скрытый параметр / ось / измерение связывает эти
            две идеи? Не пересказывай облака. Найди ось, на которой обе
            оказываются точками. Ответь одной короткой формулировкой."

       Модель возвращает текст C — скрытый параметр.

       Оценка confidence:
           centroid_A = mean(embeddings of cloud_A)
           centroid_B = mean(embeddings of cloud_B)
           sim_to_A = cosine(emb(C), centroid_A)
           sim_to_B = cosine(emb(C), centroid_B)
           confidence = min(sim_to_A, sim_to_B)

       Если confidence > 0.5 → мост найден.

    4. Если confidence < 0.5 — продолжаем цикл, расширяем облака дальше.
       С каждой итерацией у модели больше контекста для абстракции.

    5. Проверить исчерпание (novelty check):
       если новые узлы в обоих облаках начали повторяться (sim > 0.92
       с существующими) → модель исчерпала контекст → стоп

Выход:
    - success: { bridge_text: C, confidence: float,
                 cloud_a: [...], cloud_b: [...] }
    - failure: "мост не найден — A и B слишком далеки или модель исчерпала контекст"
```

### Ключевой момент: LLM как абстрактор, а не embedding similarity

Наивная версия искала бы ноду C с высокой cosine similarity к обоим облакам.
Это работает для **текстово похожих** случаев, но **упускает абстрактные связи**.

"Иммунная система" и "корпоративные банкротства" не имеют общих текстов в облаках.
Но LLM может подняться на уровень абстракции и вывести: *"адаптивность против
жёсткости в ответ на внешние угрозы"*. Этой формулировки **нет** ни в одном
облаке — она существует только как выведенная модель оси.

**Embedding нужен только для оценки confidence** найденного моста: насколько он
реально относится к обеим сторонам. Но сам поиск — через генеративную способность
LLM обобщать.

### Что важно понять

- **C не содержится в облаках.** Это новая абстракция, которую LLM выводит из
  обоих контекстов. Embedding similarity в поиске не участвует, только в оценке.
- **Облака = контекст для LLM**, а не база для поиска. Чем богаче контексты, тем
  точнее LLM найдёт общую ось.
- **Мост — это формулировка, не нода.** Её можно сохранить в граф как новую
  ноду с `collapsed_from: [A_id, B_id]` и прямыми связями к A и B.
- **Инкрементальная попытка.** Первая итерация может не дать моста (облака ещё
  малы, LLM не видит абстракции). Расширяем дальше и пробуем снова.

### Почему это мощно

- **LLM-as-abstractor** находит связи на уровне, недоступном чистым embeddings
- **Двусторонняя экспансия** даёт модели богатый контекст вместо голых двух фраз
- **Результат — текст**, сразу пригодный для чтения, сохранения и дальнейшей работы
- **Confidence на основе центроидов** — простая и честная метрика "ось реально
  общая"

### UI

- **Контекстное меню на ноде** → "Pump to..." → выбрать вторую ноду кликом
- **Визуализация**: два облака расширяются в реальном времени, при встрече
  мостовые узлы подсвечиваются, путь A → мост → B выделяется линией
- **Кнопка на результате**: "Save as synthesis" — создать collapse-ноду из моста,
  или "Verify bridge" — прогнать Smart DC на найденной связи

      возвращает путь + confidence или "not connected"

### Место в архитектуре

Накачка — **примитивная операция на узлах**, не режим. Наравне с:
- **Smart DC** (тезис/антитезис/синтез → confidence одной ноды)
- **Pump** (двусторонняя экспансия → мост между двумя нодами)

Может использоваться внутри режимов алгебры как подоперация:
- **Диспут**: накачка между противоположными позициями для поиска общей почвы
- **Турнир**: накачка между вариантами для выявления скрытых общих критериев
- **Разведка**: случайная накачка между далёкими узлами как способ серендипити

## v2: алгебра режимов

Реализация оставшихся 11 режимов из [README](README.md). Цель — один tick, 12 конфигов.

### Шаг 1: инфраструктура конфига

- [ ] **Mode config в goal-ноде** — dict с `(mode, primitive, strategy, goals, goal_type, fields)`:
  - `primitive`: `none / and / or / xor` (четыре примитива)
  - `strategy`: `unordered / seq / priority / balance` (для AND) или `comparative / dialectical` (для XOR) или `null`
  - `goal_type`: `finite / repeatable / open`
- [ ] **Goal-нода как structured object** — сейчас только text, нужны дополнительные поля
- [ ] **Stop condition framework** — функции `(graph_state) → bool | snapshot`. По типу цели:
  - finite: `confidence ≥ threshold` → RESOLVED
  - repeatable: `step_complete + trigger` → SCHEDULED
  - open: `diminishing_returns ИЛИ budget` → PARKED
- [ ] **Goal evaluation** — LLM-as-judge: "достигли цели или нет?", сравнение goal↔result

### Шаг 2: диспетчер tick

- [ ] **tick(config, graph)** — читает `primitive` + `strategy` и вызывает реализацию

Четыре примитива:
- [ ] **none** (Scout) — без целей, дивергентное блуждание
- [ ] **AND** — все цели должны быть verified
- [ ] **OR** — первая достигнутая завершает цикл
- [ ] **XOR** — выбрать ровно одну из множества

Стратегии обхода AND (четыре режима):
- [ ] **unordered** — Конструктор: любой порядок
- [ ] **SEQ** — Конвейер: по зависимостям
- [ ] **PRIORITY** — Каскад: по важности
- [ ] **BALANCE** — Весы: пропорциональная аллокация над нефинитными

Стратегии разрешения XOR (два режима):
- [ ] **comparative** — Турнир: сравнение независимых опций
- [ ] **dialectical** — Диспут: синтез противоречивых утверждений через SmartDC-on-graph

Варианты OR (по типу цели):
- [ ] **OR finite** — Гонка (первая цель завершает)
- [ ] **OR open** — Веер (уже работает как текущий цикл)

### Шаг 3: режимы single-goal по типу цели

- [ ] **Вектор** (finite) — один фокус, сходится к RESOLVED
- [ ] **Ритм** (repeatable) — heartbeat, snapshot evaluation, streak/trend
- [ ] **Горизонт** (open) — уже работает как текущий цикл

### Шаг 4: UI

- [ ] **Селектор режима** — 12 опций с подсказками
- [ ] **Динамическая форма** — поля ввода из `config.fields` под режим
- [ ] **Display целей** — goal-нода показывает структуру (список целей + оператор)
- [ ] **Snapshot для repeatable** — виджет streak/today/trend вместо sparkline сходимости

### Шаг 5: персистентность

- [ ] **State beyond session** — Ритм работает днями/неделями, Вектор — месяцами
- [ ] **History log** — timestamps, changes, confidence evolution
- [ ] **Автосохранение** — уже есть, нужна проверка для long-running режимов
