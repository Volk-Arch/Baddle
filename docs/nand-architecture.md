# NAND-архитектура — один примитив вместо всей логики

## Идея

Классический подход к «умной» системе — прописать разные ветки для
разных задач: «если это выбор, делай так; если исследование, иначе;
если противоречие, третий путь». Это работает, но это ручная логика:
каждая ветка — отдельный код, отдельная калибровка, отдельный баг.

Предложение другое: один оператор различения, `distinct(A, B)`, который
даёт число от 0 (идентичны) до 1 (максимально различаются). Вся булева
логика выводится из зон этого числа — не прописывается отдельно, а
**возникает** из значений.

Аналогия: NAND-гейт в электронике. Один элемент, из которого собирается
вся логика компьютера. `distinct()` — вероятностный NAND для мышления.

> Концепция из статьи [alphaxiv.org/abs/2603.21852v2](https://www.alphaxiv.org/abs/2603.21852v2).

---

## Два двигателя реальности

Система живёт не на 0 и 1. Она живёт на **рельефе** — пространстве
состояний, по которому скатывается к ближайшему устойчивому. 0 и 1 —
это флажки, которые мы ставим на карте когда нужно нажать кнопку.
Сам рельеф задаётся двумя независимыми принципами:

| Начало | Правило | Что измеряет | Что порождает |
|--------|---------|--------------|---------------|
| **Структура** | `A ≠ B` | Расстояние в пространстве состояний | Карту. Без неё — однородный шум |
| **Действие** | `A ≠ A` | Скорость изменения во времени | Движение по карте. Без него — застывшая схема |

Метрика расстояния — cosine, KL-дивергенция, энергетический зазор. Не
важно какая конкретно, важен принцип: **различие — базовое, тождество —
производное**.

### NAND как физика состояний

В обычной булевой логике 0 и 1 — это просто битики. В физике они — классы
состояний, и один и тот же «1» может быть разным по происхождению:

| Входы | Выход | Режим |
|-------|-------|-------|
| 0, 0 | 1 | **Rest** — тишина, система удерживается по умолчанию, нулевые затраты |
| 0, 1 | 1 | **Held** — один вход давит, порог не преодолён, система напряжена |
| 1, 1 | 0 | **Switch** — оба входа преодолели барьер → сброс/инверсия |

### 1 ≠ 1 — состояние помнит путь

Важная деталь: в реальной системе `1` от `(0,0)` и `1` от `(0,1)` — **не
одно и то же**. Путь определяет:

- **Близость к переключению.** Из `(0,1)` до 0 система дойдёт быстрее —
  она уже наполовину там.
- **Энергетическая стоимость.** Удержание 1 при `(0,1)` требует постоянного
  расхода. При `(0,0)` — нет.
- **Реакция на шум.** `(0,0) → 1` устойчиво к флуктуациям, `(0,1) → 1`
  уже «на взводе».

Это гистерезис, память пути, энергетическая история. Реальные системы
несут отпечаток того, как попали в текущее состояние. Поэтому в Baddle
различаем два подтипа «единицы»:
- `1_rest` (от 0,0) — потенциал, покой, низкий sync_error, готов
  к сканированию
- `1_held` (от 0,1) — активность, напряжение, высокая чувствительность,
  требует быстрой валидации

Сложность рождается **не из увеличения ресурсов**, а из композиции
одного правила с контекстом и историей.

### Философский корень

- `A ≠ A` — рождение наблюдения (субъект обнаружил разницу в самом себе)
- `A ≠ B` — рождение объекта/контекста (есть я и другое)
- `distinct(distinct(A,A), distinct(A,B))` — рефлексия второго порядка

## Что делает distinct

Считаем одну меру — распределение различия между входами:

- `d = distinct(A, B)` возвращает число в `[0, 1]`
- 0 — входы неразличимы
- 1 — максимально различаются

Технически: cosine distance между эмбеддингами или KL-дивергенция между
распределениями. Для графа Baddle — cosine.

**Из этого числа выводится правдоподобие.** Чем меньше различие, тем
выше согласие, тем больше вес гипотезы в байесовском обновлении.

- Правдоподобие: `L = exp(−γ · d)` — при `d → 0` имеем `L → 1` (согласие),
  при `d → 1` имеем `L → 0` (конфликт)
- Байес в лог-пространстве (стабильно, не переполняется):
  `logW_new = logW_old − γ·d + log(prior)`
- Нормализация: `W_new = softmax(logW_new / T)`, где `T` управляет
  «резкостью» выбора

`γ` — чувствительность, «резкость» правдоподобия. `T` — температура
выбора. Оба параметра **адаптивные**, не статические.

### Автокалибровка γ и T

Параметры не жёсткие. Они подстраиваются под состояние системы:

- **γ** калибруется по стабильности: если `distinct(A,A)` (похожих штук
  друг с другом) нестабильна, γ опускается (осторожнее). Стабильность
  высокая — γ поднимается (строже).
  (`γ ← clip(γ₀ / (ε + EMA(d(A,A))), γ_min, γ_max)`)
- **T** калибруется по сдвигу убеждений между тиками: большой сдвиг →
  T поднимается (мягче выбор, даём шанс альтернативам), стабильно —
  T опускается (резкий выбор).
  (`T ← T₀ · (1 + α · KL(W_t ‖ W_{t-1}))`)

### Решение, не только число

У `distinct` есть три зоны:

```
d < τ_in           → CONFIRM   (связь подтверждена, усиление)
τ_in < d < τ_out   → EXPLORE   (зона неопределённости, нужны ещё данные)
d > τ_out          → CONFLICT  (ветвление или архив)
```

Пороги `τ_in` и `τ_out` не фиксированы — они калибруются через HRV
и историю (см. ниже «HRV → γ/τ»).

## Git-коммит — версионирование каждого шага

Каждый байесовский шаг записывается как коммит с полной трассой
(структура state_graph, см. [state-graph-design.md](state-graph-design.md)):

```json
{
  "hash": "a3f8c2d1",
  "parent": "7b2e4f09",
  "timestamp": "2026-04-21T14:32:01Z",
  "node_id": 7,
  "distinct_trace": {
    "a": "embedding_hash_A",
    "b": "embedding_hash_B",
    "d": 0.342,
    "decision": "EXPLORE"
  },
  "weight_delta": { "7": 0.12, "3": -0.05 },
  "macro_applied": "XOR(A,B)",
  "gamma": 2.34,
  "temperature": 0.12,
  "thresholds": { "tau_in": 0.28, "tau_out": 0.71 },
  "physio_state": {
    "hrv_coherence": 0.73,
    "rmssd": 42,
    "stress": 0.31
  }
}
```

Что это даёт:

- **Трассировка решений.** Каждый шаг аудируем, можно посмотреть почему
  система выбрала этот вариант.
- **Автоматические ветки.** Если `d > τ_out`, создаётся ветка для
  тестирования альтернативы — система сама ищет «а что если наоборот?».
- **Revert / cherry-pick.** Можно откатить любой шаг или перенести
  вывод из ветки в main.
- **Отладка траекторий мысли.** `git log` становится хронологией мысли —
  видно как система пришла от одной идеи к другой.

## HRV → γ / τ — физиологическая калибровка

Параметры не только учатся из истории, но и дышат вместе с телом.
Сигналы тела:
- **Stress** ∝ 1/RMSSD (симпатика активна — стресс)
- **Coherence** ∝ LF/HF balance (парасимпатика синхронна — спокойствие)

Маппинг:

```
τ_in  = τ₀ · (1 + k₁ · Stress)       # стресс → шире ловим сигналы
τ_out = τ₀ · (1 − k₂ · Coherence)    # когерентность → уже горизонт
γ     = γ₀ + η · (Stress − Coherence) # стресс → строже проверка
α     = α₀ · Coherence                # когерентность → плавнее обучение
```

Интерпретация стратегий:
- **Стресс** (симпатика дома): широкий поиск + строгая фильтрация.
  Много входов проходит, но каждый жёстко проверяется.
- **Фокус** (парасимпатика): узкий горизонт + плавные обновления.
  Мало входов, мягкая калибровка.

Система не ломается от шума — она **меняет стратегию обработки**.

## Полная цепочка

```
HRV-сенсор
  → Stress, Coherence
    → задают γ, τ_in, τ_out, α
      → distinct(A, B) считает d
        → зона: CONFIRM | EXPLORE | CONFLICT
          → Bayes: W_new = W_old · exp(−γ·d)
            → Макросы: AND/OR/XOR выводятся из зон d
              → Git-коммит (d, ΔW, HRV, macro)
                → Horizon сдвигается / ветвится / архивируется
```

## Эмерджентная логика

Классические булевы операторы не хардкодятся. Они **возникают** из
зон `d`:

| Зона | Что это значит | Аналог |
|------|----------------|--------|
| `d ≈ 0` | Согласие, усиление | AND |
| `d ≈ 0.5` | Конфликт, альтернатива | XOR |
| `d ≈ 1` | Отрицание, разрыв | NOR / NOT |

Макросы формально:

```python
XOR(A, B) ≡ distinct(A, B)                               # различие = XOR
NOT(A)    ≡ 1 − weight(A)                                # или distinct(A, context)
AND(A, B) ≡ exp(−γ · (distinct(E, A) + distinct(E, B))) # оба согласны с evidence E
OR(A, B)  ≡ 1 − min(distinct(E, A), distinct(E, B))     # хотя бы один согласен
```

Система работает в **вероятностной** логике — точнее отражает реальные
когнитивные процессы и не даёт ступенчатых артефактов (когда переход
0 → 1 происходит на тысячную долю различия).

### Конфликт ≠ ошибка

Конфликт (d ≈ 0.5) — не баг. Это сигнал «оба варианта да в одном
пространстве». Система не паникует, а создаёт ветку для тестирования
альтернативы. Конфликт — источник информации, не её отсутствие.

## Pipeline — три слоя обработки

```
1. Эмбеддинги (маршрутизация)
   → distinct() считается в пространстве векторов, без текста
   → решение: CONFIRM / CONFLICT / EXPLORE

2. Символическое ядро (distinct + контекст)
   → макросы AND/OR/XOR собираются из зон d
   → Bayes update, Git-коммит, Horizon shift
   → отслеживается state_origin (1_rest vs 1_held)

3. Отложенный текст (LLM только при необходимости)
   → текст генерируется для пользователя, не для системы
   → система мыслит эмбеддингами, говорит текстом
```

---

## sync_error — главная метрика

`sync_error = d(model_prediction, user_action)` — **не абстрактная
ошибка предсказания**, а конкретная мера рассинхрона между системой и
конкретным пользователем:

- Высокий — система не понимает юзера → расширить поиск, задать вопрос
- Низкий — точное предсказание → сузить, действовать уверенно
- Обратная связь: система учится предсказывать **не мир вообще**,
  а конкретного человека

В связке с `state_origin`:
- `1_rest` (низкий sync_error) — медленные переходы, готов к сканированию
- `1_held` (высокий sync_error) — уже в работе, требует быстрой валидации

## Как это меняет Baddle

| Компонент | Сейчас | С distinct() |
|-----------|--------|--------------|
| **Граф** | Рёбра = cosine similarity + типы supports/contradicts | Рёбра = история вызовов distinct. AND/OR/XOR — макросы |
| **Байес** | `_bayesian_update(prior, p_e_h, p_e_nh)` | `logW += −γ·d + log(prior)` — одна формула |
| **Horizon** | precision → temp/top_k, 4 состояния | γ и T калибруются автоматически через d(A,A) и KL(W). Состояния = зоны d |
| **Tick dispatcher** | `if primitive == "and" / "or" / ...` | Не нужен. d определяет действие |
| **SmartDC** | 3 промпта → thesis/anti/neutral → centroid | distinct(thesis, anti) = мера конфликта. Synthesis = минимизация d |
| **Pump** | Поиск мостов через облака + LLM | distinct(cloud_A, cloud_B) → зона 0.3–0.7 = потенциальные мосты |
| **Novelty check** | Embedding similarity > threshold | distinct(new, existing) < threshold → reject |
| **Stop conditions** | `check_stop()` с 3 ветками | Все W стабилизировались (KL → 0) → stop |
| **HRV** | В планах: coherence → θ/φ | `γ = γ₀ + η · (Stress − Coherence)` — HRV напрямую крутит чувствительность |
| **Feedback** | `surprise = 1 − confidence` | `sync_error = d(prediction, user_action)` — персональная калибровка |
| **Pipeline** | Текст → парсинг → нода → эмбеддинг | Эмбеддинг (маршрутизация) → ядро → текст только для UI |

### Ключевое изменение — мышление эмбеддингами

Сейчас Baddle мыслит **текстом**: LLM генерирует строки → парсинг → нода.
Текст — промежуточный формат, и каждый шаг через LLM стоит ~2 секунды.

С distinct() система мыслит **эмбеддингами**:
- Генерация → эмбеддинг (вектор)
- distinct() считается без LLM-вызова (~1 мс)
- Текст генерируется **только для отображения** пользователю

Эмбеддинги уже лежат в нодах (`_graph["embeddings"]`). Нужно сделать их
**первичным** форматом, а текст — производным.

## Два режима мышления

Активный фокус (`CognitiveState`):
```
d < τ_in  → вход в Horizon (обработка)
d > τ_out → выход (архив)
γ, T, τ — динамические, калибруются через d(A,A) и HRV
```

Фоновый поиск (DMN-сканер):
```
Фоновый обход пар в зоне 0.3 < d < 0.7
Тестирование в изолированных ветках (git-коммит)
var(ΔW) < δ → мердж в основной граф
var(ΔW) ≥ δ → архив (не подтвердилось)
```

Переключение:
```
Высокая нагрузка → Horizon доминирует, DMN в low-priority
Покой           → DMN массовый скан, готовит инсайты
```

Бюджет вычислений делится динамически. Уже частично работает:
Scout mode = DMN, autorun = Horizon, Pump = DMN-поиск.

## Prediction error — синхронизация, не ошибка

Новая интерпретация: ошибка предсказания = **система плохо
синхронизирована с пользователем**, а не «модель плохая в вакууме».

- Большое расхождение → система не понимает юзера
- Малое → система предсказывает точно

Это расширяет текущий prediction error (`surprise = 1 − confidence`)
до персональной калибровки: система учится предсказывать не мир,
а конкретного человека.

## Априоры как распределения (а не одно число)

Сейчас prior — одна цифра (обычно 0.5). Предложение: prior =
распределение (Beta). Дополнительная информация — не только «вероятность»,
но и «уверенность в этой вероятности».

```python
# Сейчас
prior = 0.5   # "половина", и всё

# С распределениями
prior = Beta(α=2, β=3)  # "скорее нет, но не уверен"
# α > β   → скорее да
# α = β   → не знаю
# малые α,β → совсем не уверен
# большие α,β → уверен
```

`Beta(1,1)` = «понятия не имею», `Beta(50,50)` = «точно пополам, знаю
наверняка». Для 0.5 это не одно и то же состояние.

---

## Режим камеры сенсорной депривации

Автоматический режим где система **изолируется** от внешних входов:

- Отключает LLM-генерацию нового контента
- Только distinct() между существующими нодами
- Ищет скрытые паттерны в том, что уже есть

Аналог медитации: не генерировать новое, а увидеть связи в имеющемся.
Полезно когда юзер offline, а процесс всё ещё работает — не создавать
мусор, а переосмысливать накопленное.

## Шаги реализации

1. **`distinct()`** — обёртка над cosine_similarity в main.py.
   `d = (1 − cos_sim) / 2`.
2. **Байес через distinct** — заменить `_bayesian_update()` на
   `logW += −γ·d + log(prior)`.
3. **Автокалибровка γ.** EMA(d(A,A)) → γ адаптируется.
4. **Эмбеддинги первичны.** Текст генерируется только для UI.
5. **DMN с distinct.** Фоновый обход пар в зоне 0.3 < d < 0.7.
6. **HRV → γ.** `γ = γ₀ + η · (Stress − Coherence)` из готового стека.
7. **Beta-распределения для приоров.** Вместо скаляра.
8. **Камера сенсорной депривации.** Режим без LLM, только distinct().

---

## Итог

Один примитив `d = distinct(A, B)` заменяет всю ручную логику.
Мгновенно превращается в байесовское правдоподобие `L = exp(−γ·d)`.
Веса обновляются в лог-пространстве, γ и T калибруются автоматически.
Система сама балансирует фокус/поиск, версионирует каждый шаг и
устойчива к шуму без усложнения кода.

Не калькулятор, а самокалибрующийся когнитивный движок.
