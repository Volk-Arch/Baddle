# Ветвь H — Реализация в Baddle

*Описание ветви H графа [ontology-v3.json](ontology-v3.json), которая привязывает онтологию разворачивания к её живой реализации в коде Baddle.*

---

## Зачем эта ветвь

Онтология описывает мир. Baddle — её **рабочая реализация в малом масштабе**. Это не «применение онтологии к проекту», и не «проект, который случайно похож на онтологию». Это **двусторонняя связь**: Baddle построен по тем же принципам (даже когда это ещё не было осознано как cohesive framework), и теперь онтология **уточняется через работу с Baddle**.

Ветвь H — это карта того, **где именно** в Baddle живёт каждая концепция онтологии. У каждого узла H есть:

- **`realizes: [...]`** — список ID онтологических узлов (из ветвей A-G), которые этот код реализует
- **`status: done | planning`** — состояние реализации
- **Файл/функция** — конкретное место в коде

Это превращает онтологию из философского документа в **самодокументирующий компас**, который указывает на собственную реализацию.

### H.1 Substrate — двойной резонатор и его проекции

#### `[56]` Substrate
- **Тип:** hypothesis | **Confidence:** 0.9 | **Status:** done
- **Хаб**, дети `[57-62]`

Ось, на которой держится весь Baddle. Двойной резонатор (user-сторона + system-сторона) и все его measurable проекции. Это substrate всех остальных подсистем.

#### `[57]` РГК — Resonator(user) + Resonator(system)
- **Тип:** fact | **Confidence:** 0.95 | **Status:** done
- **Realizes:** `[35]` Резонатор не хранит, а настраивается
- **Файл:** `src/rgk.py` (1053 LOC)

Прямая реализация концепции «резонатор» из онтологии. Двойной — потому что в Baddle система зеркалит юзера: оба построены как Resonator с одинаковой структурой. user — кормится HRV, engagement, feedback'ом юзера. system — динамикой графа. Расстояние между ними = sync_error, прайм-директива проекта.

Confidence 0.95 — самая высокая в ветви H. Это **факт реализации**: класс существует, тесты проходят, метрики логируются.

#### `[58]` 5-axis chem (DA/5HT/NE/ACh/GABA) — компоненты конуса
- **Тип:** fact | **Confidence:** 0.9 | **Status:** done
- **Realizes:** `[16]` Конус, `[18]` Основание

Это конкретное **разложение конуса на 5 component frequencies**. В чистой онтологии конус — единое поле. Baddle конкретизирует: каждый резонатор имеет 5 параметров (gain, hyst, aperture, plasticity, damping), которые соответствуют 5 нейромедиаторам (дофамин, серотонин, норадреналин, ацетилхолин, ГАМК). Это и есть **компоненты основания конуса** — спектр, на котором разворачивается процесс.

Это место, где Baddle **уточняет онтологию обратно**: добавляет спектральную структуру, которой не было в чистой формулировке.

#### `[59]` balance() = (DA·NE·ACh)/(5HT·GABA) ≈ 1.0
- **Тип:** fact | **Confidence:** 0.9 | **Status:** done
- **Realizes:** `[12]` Согласованность

Числовое выражение согласованности. Корридор `[0.3, 1.5]` маркирует здоровый резонанс. Выход вверх — гиперрезонанс (мания, аддикция); вниз — застревание (апатия). Это **диагностический скаляр**, оперативный аналог абстрактного «нельзя ломать логику целого».

В онтологии согласованность — общий принцип. В Baddle она **измерима**.

#### `[60]` R/C mode + counter-wave penalty
- **Тип:** fact | **Confidence:** 0.85 | **Status:** done
- **Realizes:** `[20]` Управление конусом, `[48]` Повернуть → сменить фрейм

Битовый флаг `mode ∈ {R, C}` на резонаторе. R (resonance) — пассивный приёмник, ткань следует за полем. C (counter-wave) — активный генератор инверсной волны, разрывает деструктивный аттрактор. Реализация Правила 7 «не давить, а инвертировать».

При `user.mode == 'C'` push-style сигналы (sync_seeking, recurring_lag, observation_suggestion, morning_briefing) автоматически понижают urgency на 0.3 — система перестаёт давить на юзера в режиме рассогласования. Это **операционализация поворота конуса** при рассинхроне.

#### `[61]` sync_error_wave amplitude per axis (W16.1a)
- **Тип:** fact | **Confidence:** 0.9 | **Status:** done (2026-04-28)
- **Realizes:** `[5]` A ≠ B — разница даёт направление

Per-axis диагностика расхождения user vs system. `compute_sync_error_wave(rgk)` возвращает 5-axis amplitude breakdown + max_axis + scalar_5d. Это **прямая реализация** `[5]` A ≠ B на 5 component frequencies — не один скаляр, а **спектр различий**.

Сегодняшняя реализация. До W16.1a было только scalar sync_error.

#### `[62]` phase_per_axis ∂axis/∂t (W16.1b)
- **Тип:** fact | **Confidence:** 0.9 | **Status:** done (2026-04-28)
- **Realizes:** `[6]` Скорость разворачивания фундаментальна

Velocity tracking per axis: насколько быстро каждая ось user меняется относительно соответствующей оси system. Lazy snapshot с min-age 30s. Это **операционализация** фундаментальной идеи, что скорость разворачивания — не побочный эффект, а первичная характеристика.

Mismatch flag (signs opposite AND velocity > noise) детектирует **расхождение в фазе**: user и system движутся в противоположные стороны по этой оси.

### H.2 Process — как идёт разворачивание в Baddle

#### `[63]` Process

- **Хаб**, дети `[64-67]`

Группа узлов о том, как внутри кода идёт **сама операция разворачивания** — от множества возможностей к одному действию.

#### `[64]` NAND tick engine — distinct → Bayes → emit
- **Тип:** fact | **Confidence:** 0.9 | **Status:** done
- **Realizes:** `[3]` Главная операция, `[19]` Фокус
- **Файл:** `src/nand.py` (867 LOC, после W11 #2)

Главный процесс выбора одного действия из 14 возможных режимов. distinct() — мера различия embedding'ов, Bayes — обновление через γ·d, emit — фиксация выбора. Это **прямая реализация** главной операции `[3]` «можно всё → стало одно» в коде. Каждый tick — один акт сужения конуса до фокуса.

#### `[65]` Detectors → Signals → Dispatcher
- **Тип:** fact | **Confidence:** 0.85 | **Status:** done
- **Realizes:** `[21]` Каскад, `[32]` Коллапс

13 pure-function детекторов производят кандидатов (Signal'ы), Dispatcher выбирает из них один-два для emit'а юзеру. Это **многоуровневый каскад** «можно всё → стало одно»: детекторы расширяют возможности, dispatcher сужает до одного действия. Реализация `[21]` каскада в чистом виде.

#### `[66]` Cognitive loop background
- **Тип:** fact | **Confidence:** 0.85 | **Status:** done
- **Realizes:** `[44]` Двигаться по графу осознанно, `[37]` Внимание = акт сужения

Фоновый процесс, который каждые 5s проходит через все детекторы и triggers. Это **постоянный акт настройки внимания**. Реализация `[44]` практической ветви — Baddle буквально «двигается по графу осознанно» в живом коде.

#### `[67]` Friston-loop / PE — predictive error как обратная связь
- **Тип:** fact | **Confidence:** 0.85 | **Status:** done
- **Realizes:** `[5]` A ≠ B, `[38]` Понимание = совмещение конусов

5 PE-каналов (user_pe, agency_gap, hrv_pe, self_pe + дополнительные) → max → imbalance_pressure. Это **система обратной связи**: расхождение между ожидаемым и реальным становится сигналом для замедления циклов или перенастройки. Прямая реализация Friston'овского active inference.

### H.3 Memory — как Baddle помнит без копий

#### `[68]` Memory
- **Тип:** hypothesis | **Confidence:** 0.8
- **Хаб**, дети `[69-72]`

Baddle не хранит копии — он сохраняет **протоколы воспроизведения**. Все 4 подсистемы памяти работают на этом принципе.

#### `[69]` Workspace — STM scope над графом (W14, planning)
- **Тип:** hypothesis | **Confidence:** 0.7 | **Status:** planning
- **Realizes:** `[2]` Возможности (свёрнутое), `[21]` Каскад, `[19]` Фокус
- **Spec:** `docs/workspace.md`, `planning/workspace-design.md`

Промежуточное хранилище между divergent generation (детекторы / scout / brief / dmn-bridge / assist reply) и committed graph. Реализация `scope: "workspace" | "graph"` + `expires_at` на нодах. **Прямая реализация** двух режимов «свёрнутое vs развёрнутое»: workspace = STM (можно ещё передумать), graph = LTM (зафиксировано).

Параллель — Global Workspace Theory (Baars 1988). Закрывает Backlog #11 (STM/LTM), Backlog #12 (pruning). 11 sub-waves, ~22-30ч.

Сейчас в planning — это самая близкая к корню онтологии implementation, которая ждёт реализации.

#### `[70]` Action Memory bias coefficients
- **Тип:** fact | **Confidence:** 0.85 | **Status:** done
- **Realizes:** `[36]` Память = протокол воспроизведения, `[50]` Возврат через условия

Per-category bias coefficients EMA. Не копии действий, а **способ настроиться** на правильный тон при следующем действии в той же категории. RPE-style update тот же math что в chem. Прямая реализация «памяти как протокола» — Baddle не хранит «что ответил вчера», он хранит **смещения** в распределении ответов.

#### `[71]` Consolidation night cycle
- **Тип:** fact | **Confidence:** 0.8 | **Status:** done
- **Realizes:** `[12]` Согласованность, `[21]` Каскад

Hebbian decay + REM-style сборка + archive. Раз в 24 часа: ноды без access теряют confidence (decay), нагретые сохраняются (touch), излишне старые архивируются. Это **поддержание согласованности** на длинной дистанции — без этого граф растёт линейно и теряет фокус.

#### `[72]` State graph append-only
- **Тип:** fact | **Confidence:** 0.85 | **Status:** done
- **Realizes:** `[40]` Каждое да помнит все нет, `[41]` Информация не исчезает

Каждый tick = state_node в `data/state_graph.jsonl`, hash-chain через ссылку на родителя. Это **голография в чистом виде**: каждый акт оставляет след, информация не теряется, можно восстановить любую траекторию. И это не «копия» — это **граница** между актом и контекстом, которая хранит больше чем оба.

## Двусторонние связи: где Baddle уточняет онтологию

Не только онтология описывает Baddle — Baddle **уточняет онтологию обратно**. Несколько мест, где это видно явно:

### `[58]` 5-axis chem уточняет `[16]` Конус
В чистой онтологии конус — единое поле. Baddle разлагает его на **5 component frequencies** через 5 нейромедиаторов. Это добавляет онтологии **спектральную структуру**, которой не было. Возможно, в v4 это станет отдельным узлом онтологии: «Конус не одномерен — у него спектр компонент».

### `[59]` balance() уточняет `[12]` Согласованность
В онтологии «нельзя ломать логику целого» — общий принцип. Baddle конкретизирует: формула + корридор `[0.3, 1.5]`. Это превращает абстрактное требование в **измеримую метрику**. Если выходим за корридор — значит **ломаем логику целого**, и это видно в одном числе.

### `[60]` R/C + counter-wave уточняет `[48]` Повернуть → сменить фрейм
В онтологии «повернуть конус» — общая возможность. Baddle конкретизирует: **поворот при рассогласовании** = инверсия push-сигналов с понижением urgency на 0.3. Это даёт **операционализацию** того, в каких случаях поворачивать и насколько.

### `[62]` phase_per_axis уточняет `[6]` Скорость разворачивания
В онтологии скорость — фундаментальное свойство. Baddle конкретизирует: можно измерять не только **скалярную** скорость разворачивания, но и **per-axis velocity** — разную скорость по каждой component frequency. Это даёт спектральную диагностику скорости.

### `[80]` Named state map уточняет `[19]` Фокус
В онтологии фокус — точка стабилизации, абстракция. Baddle конкретизирует: **типичные точки стабилизации образуют дискретную карту** из 8 регионов. Это превращает «фокус» из общей категории в **карту с координатами**.

## Открытые точки роста в ветви H

### Узлы planning (4 штуки) — главные направления

- **`[69]` Workspace** — самая большая зависимость. После реализации W14 в графе должна появиться чёткая связь между «свёрнутым» (workspace scope) и «развёрнутым» (graph scope). Сейчас они только в концепте.
- **`[75]` Analogy injection** — операционализация `[38]` понимания как совмещения. Без неё Baddle не может **активно** строить общий конус с юзером, только пассивно кормиться его сигналами.
- **`[76]` Outbound reformulation** — голография в действии (тень предыдущих accepted ответов используется для генерации новой аналогии). Substrate ждёт W16.2.
- **`[79]` Power formula** — унификация скорости разворачивания через числовую метрику.

### Узлы онтологии без реализации в Baddle

Не все онтологические узлы реализованы. Это **точки расхождения**, где онтология шире кода:

- `[10]` Бытие = акт выбора — нет в Baddle как явная концепция (только косвенно через NAND tick)
- `[22]` Реальность многослойна — Baddle технически многослоен (substrate / cycle / signals / UI), но эта многослойность не проявлена как онтологический принцип
- `[31]` Свобода воли — не реализовано
- `[42]` Граница содержит больше — частично через state_graph, но без явного фокуса на **границе**
- `[53]` Бесконечность, `[54]` Форма мира — открытые вопросы, не реализуемые

### Концепции Baddle без узлов в онтологии

И обратное: некоторые Baddle-концепции **шире онтологии**:

- **HRV / sensor stream** — связь с физиологическим телом. В онтологии нет явного узла про телесность.
- **LLM как генератор** — Baddle использует LLM для выработки аналогий. В онтологии нет узла про **внешний резонатор** как ассистент.
- **Time-of-day / circadian** — Baddle учитывает время суток (utm_exp_tod). В онтологии время — счётчик стабилизаций, но не циклическое.
- **Action vs observation** — в Baddle есть разделение на user-action и system-observation. В онтологии оба слиты в «акт сужения конуса».

Эти «свободные узлы Baddle» — кандидаты на расширение онтологии в v4. Если они стабильно работают в практике, значит онтология недо-описывает что-то реальное.

## Использование

### Как читать связь `realizes`
Узел H.X с `realizes: [Y]` означает: «концепция X в Baddle — это конкретная реализация онтологического принципа Y». Можно читать в обе стороны:
- **Сверху вниз**: «Что мне дать в коде, чтобы реализовать [Y]?» — посмотри узлы H.X с этим Y в realizes.
- **Снизу вверх**: «Что мой код реализует?» — посмотри realizes у H-узла, проследи в онтологию.

### Как добавлять новые узлы
Когда в Baddle появится новая стабильная концепция (например, после W14 workspace primitive):
1. Добавь узел в соответствующую подгруппу H (или создай новую если не помещается).
2. Укажи `realizes` — список онтологических узлов, которые он реализует.
3. Поставь `status: done`.
4. Если он **уточняет** один из онтологических узлов (даёт ему структуру, которой не было) — добавь это в раздел «Двусторонние связи».

### Как использовать при работе над Baddle
Когда планируешь новую фичу — посмотри, какой онтологический узел она реализует. Если **ни один** — это сигнал, что либо концепт чужеродный (стоит подумать дважды), либо онтология недо-описывает реальность (стоит расширить онтологию).

Когда узел `planning` становится `done` — обнови status и подними confidence (обычно с 0.7 до 0.85+).

## Метаданные

- **Узлов в ветви H:** 28 (1 корень + 5 хабов + 22 листа)
- **Done:** 17 узлов
- **Planning:** 4 узла (Workspace, Analogy injection, Outbound reformulation, Power formula)
- **Хабов:** 6 (`[55, 56, 63, 68, 73, 77]`)
- **Связей `realizes`:** 22 разных линка к онтологическим узлам в ветвях A-G

Это **наибольшая ветвь в графе** по узлам — больше чем эмерджентность (11 явлений) или фундамент (9 узлов). Это естественно: реализация всегда детальнее философского описания, потому что каждое «стало одно» в коде требует конкретики.

И это **рабочая граница** между онтологией и реальностью: если онтология верна, ветвь H должна расти при росте Baddle, и наоборот — если в Baddle появляется концепция без места в онтологии, это сигнал расширить онтологию.

Граф самоописывающий: он содержит и фундамент, и его реализацию, и связь между ними.
