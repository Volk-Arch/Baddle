# Дыхательный режим — Слой 4

Фича для пользователя: Baddle может предложить дыхательную сессию
(coherent breathing 5→5 или box 4-4-4-4) когда видит рассогласование
между состоянием и типом активности. Self-guided, не лечит.

Статус: **черновик для обсуждения**. Блокер `frequency_regime` снят
2026-04-25 (реализован как `UserState.frequency_regime` derived property).
Решение об имплементации — после того как frequency_regime покажет
стабильные данные на 1-2 нед реального use.

---

## Проблема

У Baddle есть HRV-данные (с Polar) и nейрохимия. Он может видеть:
пользователь собирается сесть за heavy task, а его ВНС в симпатике
(короткая волна) — физически не готов к длинной волне ассоциативного
мышления. Или наоборот: пришло время отдыха, а HRV показывает
активный режим.

Сейчас Baddle **не вмешивается** в это — только показывает
neurochem-бары и cone-viz. Юзер должен сам догадаться что
«HRV низкая → надо бы дышать».

## Резонансная рамка

Дыхание — **ручка настройки несущей частоты организма**. Это не
вопрос «расслабления» — это физическое изменение длины волны
через активацию ВНС:

- **Coherent breathing (5→5)** — 6 циклов/мин, резонанс с
  барорефлексами, активация парасимпатики → удлинение волны,
  рост HRV, синхронизация сердце-мозг. Режим для ассоциативного,
  стратегического мышления.
- **Box breathing (4-4-4-4)** — задержки стабилизируют симпатическую
  активацию. Короткая волна, острый фокус, подготовка к тактике.
- **4-7-8** (вдох 4 — задержка 7 — выдох 8) — экстренный parasympathetic
  reset при панике/тревоге.

Baddle предлагает тот режим, который физически подходит под
ближайшую задачу или текущее рассогласование.

## Спецификация

### Когда предлагать

Proactive check в [cognitive_loop.py](../src/cognitive_loop.py) —
новый `_check_breathing_suggestion` (22-й фоновый check):

**Триггеры предложения:**

1. **Mismatch текущего и планируемого.** Через 10+ мин наступает
   heavy task (calendar / goal с высокой сложностью) И frequency_regime
   сейчас `short_wave` → предложить coherent breathing для перехода
   в long_wave.

2. **Mismatch отдыха.** Текущее время в rest-zone (календарь / evening)
   но frequency_regime `short_wave` ≥ 2 часов → предложить
   coherent для активации recovery.

3. **Высокий focus_residue.** Если residue > 0.7 → предложить
   box breathing или 4-7-8 для сброса когнитивного шума.

4. **User-triggered.** Чат-команда `/dyhanie` или кнопка в
   morning briefing.

**Throttle:** не более 1 предложения в 3 часа. После принятого
или отклонённого предложения — next-window 6 часов (не навязывать).

### Как предлагать

**Ни в коем случае не:**

- ❌ «У тебя плохое состояние»
- ❌ «Ты в стрессе, подыши»
- ❌ «Твоя HRV опасно низкая»
- ❌ Не называть эмоций, не диагностировать

**Можно:**

- ✅ «Впереди плотная работа — 3 цикла дыхания 5→5 дают 5 минут
  настройки. Попробуешь?»
- ✅ «Резонанс с задачей сейчас низкий. Небольшая калибровка
  через дыхание возможна — но только если хочешь.»
- ✅ «[кнопка] 🫁 Синхронизироваться (3 мин)»

Текст генерируется LLM через существующий `suggestions.py` →
mode `breathing_suggestion` с контекстом (какой режим, почему).

### UI-взаимодействие

**Alert-card** в чате (как existing observation_suggestion):
- Иконка 🫁
- Краткое обоснование (одна строка)
- Кнопка `Начать` → открывает модальный overlay
- Кнопка `Позже` → throttle на 3 часа
- Кнопка `Не предлагать сегодня` → скрыть этот check до завтра

**Модальный overlay дыхания:**
- Анимированный круг (expanding/contracting) синхронно с ритмом
- Счётчик циклов (например, 3 из 5)
- Текст: `Вдох...` / `Выдох...` / `Задержка...`
- Кнопка `Закончить` — останавливает, записывает partial в graph
- После завершения — записывается как user action `breathing_completed`
  с outcome (изменение HRV за следующие 5 мин через
  action-memory cycle)

### Что записывается в граф

Action node:
```
{
  "action_kind": "breathing_suggestion",
  "actor": "baddle",
  "context": {
    "regime": "coherent_5_5",
    "trigger": "upcoming_heavy_task",
    "frequency_before": "short_wave"
  }
}
```

Outcome node (через 5-10 мин):
```
{
  "action_kind": "breathing_outcome",
  "context": {
    "regime_before": "short_wave",
    "regime_after": "long_wave",
    "hrv_coherence_delta": +0.2,
    "delta_sync_error": -0.05
  }
}
```

Action Memory цикл закрывается: Baddle учится когда
предложения реально помогают (snu_sync улучшается) и когда
нет — и адаптирует scoring через `score_action_candidates`.

### Критерий ценности

Через 2 месяца смотрим:
- **Принятие:** accepted / (accepted + declined). Если < 20% —
  юзер считает предложения бесполезными или инвазивными.
- **Эффект:** средний `delta_sync_error` в outcome — отрицательный
  (sync_error падает) = фича работает по прайм-директиве. Близкий
  к 0 = не помогает физически.
- **False positives:** accepted но outcome не улучшился — проблема
  в триггере (предложили не там где нужно).

## Граница «не лечит»

Baddle **не**:
- диагностирует состояние
- говорит «ты в тревоге / депрессии / стрессе»
- требует выполнения
- отслеживает compliance / streak'и
- давит метриками «ты пропустил 3 дня»

Baddle **предлагает** физиологический инструмент с явным
obоснованием (frequency mismatch). Решение — за пользователем.
Одно «Не предлагать сегодня» → тишина до завтра. 3 отклонения
подряд на одном триггере → учиться (scoring) не предлагать
этот тип этому пользователю.

Это согласуется с [world-model.md](../docs/world-model.md) —
«Baddle не продуктивный любой ценой». Дыхательный режим тоже
опциональный, не ritual.

## Менее инвазивные альтернативы

Если proactive invite оказывается слишком invasive — fallback-варианты:

1. **Self-guided only.** Нет proactive check, только ссылка в
   morning briefing: «Если чувствуешь рассогласование — 🫁 доступно
   в боковой панели».
2. **Indicator without prompt.** В header показывается индикатор
   frequency_regime (🔵/🔴) — юзер сам решает нужна ли калибровка.
   Клик открывает breathing overlay.
3. **Смешанный.** Indicator всегда + proactive prompt только при
   очень сильном mismatch (frequency_regime ≠ task requirement
   И accepted history > 50%).

Решение о степени проактивности — **после** того как
frequency_regime покажет стабильность порогов (1-2 нед реальных
данных).

## Оценка

- **Backend:** `_check_breathing_suggestion` + новая suggestion-mode
  ~2ч.
- **UI:** модальный overlay с анимацией + записи в граф ~2ч.
- **Integration с action-memory:** outcome detection + scoring
  расширение ~1ч.
- **Итого:** ~5ч, средний риск (новый взаимодействие-паттерн).

## Зависимости

`frequency_regime` derived field — ✅ реализован 2026-04-25 как
`UserState.frequency_regime` (см. `src/user_state.py`). Возвращает
`long_wave` / `short_wave` / `mixed` / `flat`. Без него
`_check_breathing_suggestion` не смог бы правильно триггерить —
блокер снят.
